# HEARTBEAT — 주기적 자동 실행 작업 및 타임아웃 복구 시스템

## 🔧 타임아웃 복구 전략
OpenClaw에서 LLM 요청 시 빈번한 타임아웃이 발생할 경우를 대비한 자동 복구 메커니즘

### ⚠️ 타임아�트 감지 시 자동 복구 절차

#### 연속 타임아웃 감지
**조건:** zai:default 모델 연속 2회 타임아웃 발생 시  
**실행 작업:**
1. Slack 경고 발송 (재영님에게 알림)
2. fallback 모델로 전환 (zai/glm-4.5-flash → zai/glm-5)
3. 세션 상태 검증 및 복구 시도

#### 세션 고정 상태 감지
**조건:** 세션 응답 없음이 5분 이상 지속 시  
**실행 작업:**
1. 세션 리셋 시도 (새로운 세션 생성)
2. 이전 작업 상태 백업 및 복구
3. 재시도 횟수 제한 (최대 3회)

#### 메모리 누수 감지
**조건:** 대화 히스토리 10턴 이상 누적 시  
**실행 작업:**
1. memory-compactor 호출 (메모리 정리)
2. 불필요한 캐시 삭제
3. 세션 상태 최적화

## ⏰ 주기적 자동 실행 작업

### 매주 월요일 08:00 KST
- **뉴스레터 자동 발행:** `newsletter-generator` 스킬 실행
  - cron: `0 8 * * 1` (Asia/Seoul)
  - 흐름: 트렌드 수집 → AI 에피소드 생성 → Slack 승인 요청 → 이메일 발송
  - 설정: `~/.openclaw/cron/jobs.json`
  - 스킬: `~/.openclaw/skills/newsletter-generator/`

### 매 30분
- **대화 히스토리 정리:** 10턴 이상이면 memory-compactor 호출
- **세션 상태 점검:** 활성 세션 상태 검증
- **API 사용량 모니터링:** 토큰 사용량 점검

### 매 1시간
- **타임아웃 통계 분석:** 연속 타임아웃 패턴 분석
- **모델 성능 평가:** 응답 시간 및 정확도 측정
- **자동 복구 트리거 확인:** 복구 메커니즘 작동 상태 확인

### 매 12시간
- **전체 세션 상태 점검:** 모든 활성 세션 상태 검증
- **캐시 시스템 정리:** 오래된 캐시 데이터 삭제
- **성능 최적화:** 시스템 리소스 최적화

### 매일 자정
- **일일 리포트 생성:** 작업 완료률, 문제 발생률 통계
- **백업 작업:** 세션 상태 및 데이터 백업
- **시스템 상태 리포트:** 전체 시스템 건강 상태 확인

## 🚨 예외 상태 처리

### 치명적 오류 시
**동작:** 즉시 Slack 알림 + 자동 복구 시도  
**내용:**
- 세션 완전 종료 시 새 세션 생성
- 데이터 손실 방지를 위한 백업 복구
- 사용자에게 상황 안내

### 네트워크 장애 시
**동작:** 재연결 로직 실행  
**내용:**
- 30초 간격으로 재연결 시도 (최대 5회)
- 오프라인 모드 활성화
- 네트워크 복구 후 자동 동기화
